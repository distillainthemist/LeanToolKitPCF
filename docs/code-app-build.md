# LeanToolKit code app ŌĆö build & deploy

How to develop, test, and ship the `app/` Power Apps code app. Design of
record: [code-app-plan.md](code-app-plan.md). The canvas-era build kit
([board-app-build.md](board-app-build.md)) is historical.

## Prerequisites

- **Node 22+** ŌĆö the `pac code` toolchain floor. On this Mac it's a
  Homebrew keg: prefix commands with
  `export PATH="/opt/homebrew/opt/node@22/bin:$PATH"`.
- **pac CLI 2.9.3+** (`dotnet tool install --global
  Microsoft.PowerApps.CLI.Tool`). 2.8.x crashes on macOS constructing
  the Windows-only MSAL broker.
- Auth: `pac auth create --deviceCode`, signed in as a maker in the
  target environment. Check with `pac org who`.
- The environment needs **Power Apps Code Apps** enabled (PPAC ŌåÆ
  environment ŌåÆ Settings ŌåÆ Product ŌåÆ Features). The toggle takes
  ~20ŌĆō25 minutes to propagate; pushes 403 with
  `CodeAppOperationNotAllowedInEnvironment` until then.

## Local development

```sh
cd app
npm ci
npm run dev        # vite on :5180 ŌĆö DEMO MODE (no host, no Dataverse)
npx tsc -p tsconfig.json
npx vitest run     # store unit tests (pure logic; no host needed)
```

Demo mode shows banner'd screens with canned data; everything Dataverse
(board screen, composer, card editor) needs the deployed app.

## Push to the environment

```sh
cd app
npm run build      # vite ŌåÆ dist/  (base "./" is load-bearing ŌĆö the
                   # appruntime serves from a deep path; absolute
                   # /assets URLs 404 and the app renders blank)
pac code push      # updates the app in power.config.json's environment
```

- `power.config.json` pins `appId` + `environmentId`; `--environment`
  on push does **not** retarget ŌĆö edit the config to point elsewhere.
- Open sessions show "You're using an old version ŌĆö Refresh" after a
  push; new sessions get the latest.
- `pac code push --solutionName LeanToolKitData` associates the app
  with the solution **only when the push creates the app** (no appId).
  Updates silently skip association. The Dev app predates the solution;
  to move it inside, delete the app in the maker portal and push once
  with `--solutionName` (new app id ŌåÆ new play URL).

## Schema (Dataverse tables)

Declarative source of truth: `data/schema.mjs`, deployed idempotently:

```sh
node data/get-token.mjs                       # device-code ŌåÆ token file
node data/deploy-schema.mjs <env-url> <file>  # create/update tables
node data/verify-schema.mjs <env-url> <file>  # read-back check
```

Everything lands in the **LeanToolKitData** solution (`ben` publisher).
Ship it managed:

```sh
pac solution export --name LeanToolKitData --managed \
  --path data/dist/LeanToolKitData_managed.zip
pac solution import --path data/dist/LeanToolKitData_managed.zip \
  --environment <prod url>
```

`data/dist/` is git-ignored ŌĆö exports are build artifacts.

## Typed Dataverse clients

`app/src/generated/` (models + services) is **committed source**,
generated by `pac code add-data-source -a dataverse -t <entity set>`.
Regenerate only when the schema changes, then re-run the type-check ŌĆö
the store's `upsertWhere` relies on the services' create/update shapes.

## CI/CD

- **App CI** (`.github/workflows/app-ci.yml`): type-check + vitest +
  build on every push/PR touching `app/`, `controls/`, or `shared/`.
- **Deploy app** (`.github/workflows/deploy-app.yml`, manual):
  exports the managed LeanToolKitData solution from Dev, imports it
  into Production, rebuilds the app, retargets `power.config.json`,
  and `pac code push --solutionName LeanToolKitData`. The first prod
  push creates the app inside the solution ŌĆö record the printed app id
  as the `PP_PROD_APP_ID` repo variable so later runs update in place.

One-time setup for the pipeline:

```sh
# one Entra app registration + application user in BOTH environments
pac admin create-service-principal --environment <dev url>
pac admin create-service-principal --environment <prod url>
```

Repo **secrets**: `PP_TENANT_ID`, `PP_APP_ID`, `PP_CLIENT_SECRET`.
Repo **variables**: `PP_DEV_URL`, `PP_PROD_URL`, `PP_PROD_ENV_ID`,
`PP_PROD_APP_ID` (blank until the first prod push).

## Gotchas collected along the way

- vite `base: "./"` ŌĆö see push section; never remove it.
- Node 22 for anything `pac code`; the PCF toolchain stays on Node 20.
- The wizard blob nests scheduler config under `config.` ŌĆö external
  patches to occurrence settings must write there.
- Host caches pushed bundles per session, not data: Dataverse reads are
  live (verified during the pilot).
